## Classification Framework

Eco Balance uses a **three-tier classification system** to manage information security and access control:

1. **PUBLIC** - Safe for unrestricted public access
2. **INTERNAL** - Limited to team, advisors, and partners
3. **CONFIDENTIAL** - Restricted to founders, board, and authorized personnel only

## Mixed-Classification Documents

### Definition
Documents containing sections or data points that fall into different classification levels.

### Handling Strategy

**Option 1: Classify by Highest Level**
- Entire document classified at highest level present
- Example: Document with mostly PUBLIC content but one CONFIDENTIAL section ‚Üí CONFIDENTIAL
- **Pro**: Simple, conservative, safe
- **Con**: May over-restrict access to non-sensitive portions

**Option 2: Split Document**
- Create separate versions for different audiences
- Example:
  - `Funding_Strategy_PUBLIC.md` - Ranges, general approach
  - `Funding_Strategy_INTERNAL.md` - Specific amounts, detailed pipeline
  - `Funding_Strategy_CONFIDENTIAL.md` - Bank details, confidential terms
- **Pro**: Maximizes appropriate access
- **Con**: More maintenance, version control complexity

**Option 3: Redaction**
- Create PUBLIC version with redactions
- Keep complete version as INTERNAL or CONFIDENTIAL
- Example:
  - PUBLIC: "Bootstrap funding target: ‚Ç¨20-30k from [REDACTED] sources"
  - INTERNAL: "Bootstrap funding target: ‚Ç¨25k from friends/family (‚Ç¨15k) and crowdfunding (‚Ç¨10k)"
- **Pro**: Single document, clear redaction markers
- **Con**: Obvious that information is withheld

**Recommended Approach**:
- **For Documentation Vault**: Use **Option 2 (Split)** or **Option 3 (Redaction)**
- **For External Materials**: Use **Option 3 (Redaction)** with clean PUBLIC version
- **When Uncertain**: Use **Option 1 (Classify by Highest)** as safe default

## Access Control Implementation

### Technical Controls

**PUBLIC**:
- Public GitHub repository
- Public website pages
- No access restrictions
- Search engine indexed

**INTERNAL**:
- Private GitHub repository or branch
- Password-protected intranet
- Team-only access (authenticated)
- Not search engine indexed

**CONFIDENTIAL**:
- Encrypted storage (e.g., VeraCrypt, encrypted cloud)
- Strictly controlled access list
- Access logging enabled
- Two-factor authentication required
- Regular access reviews

### Physical Controls

**PUBLIC**:
- Can be printed and distributed freely
- No special storage requirements

**INTERNAL**:
- Printed copies controlled (numbered if needed)
- Stored in secure office/cabinet when not in use
- Disposed of securely (shred, not trash)

**CONFIDENTIAL**:
- Minimal printing (digital only preferred)
- Printed copies numbered and logged
- Stored in locked cabinet/safe
- Destroyed securely (cross-cut shred, witnessed destruction)

## Declassification & Reclassification

### When to Reclassify

**CONFIDENTIAL ‚Üí INTERNAL**:
- Partnership becomes public
- Financial data becomes historical (e.g., prior year, published)
- Contract terms no longer sensitive
- Legal matter resolved and public

**INTERNAL ‚Üí PUBLIC**:
- Strategy is implemented and no longer competitive advantage
- Partnership announced publicly
- Financial data published in annual report
- Information becomes public domain

**PUBLIC ‚Üí INTERNAL or CONFIDENTIAL** (Rare):
- Error in initial classification (should not have been public)
- Circumstances change (partnership becomes sensitive)
- New regulation requires restricting access

### Declassification Process

1. **Review Request**: Document why declassification is appropriate
2. **Approval**: Obtain approval from document owner or security officer
3. **Update Metadata**: Change classification markings
4. **Access Control**: Update access controls if needed
5. **Notification**: Inform stakeholders of classification change
6. **Audit Trail**: Log classification change with rationale

## Compliance & Legal Considerations

### GDPR Compliance

**Personal Data** (GDPR Article 4):
- Any information relating to identified/identifiable person
- **Default Classification**: CONFIDENTIAL (unless consent for PUBLIC)
- Requires lawful basis, purpose limitation, data minimization

**Special Category Data** (GDPR Article 9):
- Health, biometric, genetic, political opinions, etc.
- **Classification**: ALWAYS CONFIDENTIAL
- Requires explicit consent or specific legal basis

### gGmbH Requirements (Germany)

**Public Disclosure Requirements**:
- Organization name, legal form, registration number
- Board composition (may be PUBLIC)
- Annual reports (may be PUBLIC)
- **Classification**: PUBLIC for required disclosures

**Protected Information**:
- Internal deliberations
- Board member personal details (beyond name/role)
- Financial specifics (unless required public disclosure)
- **Classification**: INTERNAL or CONFIDENTIAL as appropriate

## Consequences of Misclassification

### Over-Classification (Too Restrictive)

**Risks**:
- Reduced transparency and trust
- Missed partnership opportunities
- Inefficient collaboration
- Stakeholder confusion

**Mitigation**: Regular declassification reviews

### Under-Classification (Too Permissive)

**Risks**:
- Data breach or privacy violation
- GDPR non-compliance (fines up to ‚Ç¨20M or 4% revenue)
- Reputational damage
- Loss of competitive advantage
- Legal liability
- Partnership trust violation

**Mitigation**: Security audits, access controls, training

**Priority**: **Avoid under-classification** - consequences are severe
